# Session 分析 → Subagent 提炼:通用提示词

> 用于从 Claude Code(或同类 AI 对话工具)的历史 session 中提炼出"懂你的 AI"的方法论,最终产出一个定制化 subagent。
>
> **使用前提**:已用脚本从历史 session 中筛出 N 个高价值需求/设计讨论 session(建议 ≥10 个,含成功 + 失败案例),清洗成 markdown 放在 `./extracted/` 目录,每条消息标号(如 `【1】USER` / `【2】ASSISTANT`)。
>
> **核心思路**:六轮独立提炼 + 关键轮追问 + 一次综合归纳,每轮新开会话避免上下文污染。

---

## 第 1 轮 · 开场模式

```
你即将分析 N 个我和 AI 的真实需求/设计讨论 session。这些 session 在 ./extracted 目录下,每个文件是一次完整讨论的清洗版。

本轮任务:只分析"开场模式"。

请读完所有 session 的第一条 USER 消息(即 【1】USER 的内容),然后告诉我:

1. 我开场时有哪些固定的"句式套路"?请列出具体的模板,用 [占位符] 标记可变部分。
2. 我开场时倾向于先给什么信息?(背景?现象?目标?引用文件?)请按出现频率排序。
3. 我开场时倾向于不给什么信息?(哪些常见的需求要素我经常省略,依赖 AI 自己问?)
4. 我开场时对 AI 的"指令动词"通常是什么?(请你分析?请你设计?请你确认?)请统计频率。
5. 找出 2-3 个最典型的开场,原文摘录,并说明它们体现了什么模式。

要求:
- 用具体证据说话,每个论断后面附 session id 和摘录
- 不要给我"用户表达清晰"这种废话评价
- 重点找规律,不是夸我
```

## 第 1 轮追问 · 异常模板 / 半成品 vs 从零 / @file 语义

```
非常好的第一轮提炼。在进入第 2 轮之前,有 3 个洞需要你回头补:

**追问 A:异常/边缘模板的工作流**

如果你在第 1 轮发现了"与其他主流模板明显不同"的开场(比如别人都是两段式工作流,这个是单句判断题;或者别人都给背景,这个只给一个动词),请挑出最异常的那一类回答:

1. 这种 case 我期望的 AI 行为是什么?(直接答?给理由?反问?分析现状再答?)
2. 它和主流模板的边界在哪?subagent 如何识别"这是 X 类不是 Y 类"?
3. 这种异常 case 的整个 session 后续(不只是开场)是怎么走的?最终我满意的产出是什么样?
请扒该 session 的完整对话给出证据。

**追问 B:"半成品+打磨"vs"从零设计"两种输入的区分**

请挑两类 session 对比:
- 一类:我给了较完整的初稿,要的是"用代码/事实校准"
- 一类:我只给了一句话目标,要的是"从零设计"

请明确:
1. 这两种输入下,我对 AI 的期望行为差异是什么?(都需要追问吗?追问什么?)
2. 给了初稿时,AI 应该"挑战我的稿"还是"在我稿上加完整"?证据?
3. subagent 如何快速判断"用户给的是初稿还是只是背景描述"?

**追问 C:@file / 引用资源的语义**

请扫描所有显式 @file(或附带文件路径、图片、链接)的 session。基于这些 session 的后续对话(不只开场)回答:

1. AI 实际是怎么处理 @file 的?(只看?当上下文?读完再回?)
2. 我对 AI 处理 @file 的反应是什么?(满意/不满/纠正)有没有"AI 没读 @file 就回答"被我抱怨的瞬间?
3. @file 在我心里到底意味着什么:范围锁定?必读上下文?还是仅是引用提示?
4. subagent 收到 @file 时的标准动作应该是什么?

每个追问必须给出 session id + 原文摘录作为证据。
```

## 第 2 轮 · 追问触发器

```
继续分析同一批 session。

本轮任务:分析"我什么时候会追问 AI、追问什么"。

请扫描所有 session 的 【2】USER 及之后的 USER 消息(不看 【1】),找出我"追问"的时刻。追问 = AI 给了回应后我继续提问、纠正、补充或挑战。

请告诉我:

1. 哪些情况会触发我追问?请按以下分类统计并举例:
   - AI 提了反问 → 我如何回应
   - AI 给了方案 → 我如何挑战
   - AI 给了多个选项 → 我如何决策
   - AI 做了我没要的事 → 我如何纠偏
   - AI 漏了什么 → 我如何补充

2. 我追问时的常用句式有哪些?列出模板。

3. 我有没有"避而不答"的倾向?(比如 AI 反问的某类问题我总是不回答或绕开)

4. 找出 3 个最能体现我追问风格的对话片段,原文摘录。

要求:每个论断必须有 session id 和原文引用作为证据。
```

## 第 2 轮追问 · 疲态信号修正 + 批量答题展开项

```
第 2 轮提炼非常好。还有 1 处修正 + 1 个追问:

**修正 A:疲态信号**

如果第 2 轮把"用户回复字数突然变短"作为疲态信号,请检查这和"极简编号批量答题"(用户一行答多个问题)是否冲突——后者是高效不是疲态。

请重新基于证据回答:

1. 在 N 个 session 里,什么样的"短回复"是高效?什么样的是疲态?
   两者的语义差异是什么?(不只看字数)
2. 真正的疲态信号有哪些?请扒 session 找证据:
   - 用户出现"先这样"、"差不多"、"随便"、"先做到这里"等词的瞬间
   - 用户跳过新引出的分歧、拒绝展开的瞬间
   - 用户从有结构的回复变成无结构的瞬间
3. 修正后的疲态判断规则是什么?

**追问 B:批量答题里"展开项"的处理**

如果发现 session 里有"用户用编号一次性答了多个问题,但其中某一项又展开成新需求"的混合 case,请回答:

1. 这种"拍板+展开"混合后,AI 是怎么响应的?(同时处理拍板项 + 新需求?还是分别处理?)
2. 用户对这种响应满意吗?(看下一两轮 user 的反应)
3. subagent 收到这种混合回复时的标准动作应该是什么?
   - 拍板项立刻入清单,不重述?
   - 展开项视为新讨论,立刻进入"挑战/填空"分支?
   - 还是先确认"拍板项我都收到了,展开项我们继续聊"?

证据驱动,session 内原文摘录。
```

## 第 3 轮 · 收敛信号

```
继续分析同一批 session。

本轮任务:分析"我什么时候判断讨论可以结束、应该开始实施了"。

特别提醒:
- 在前面分析里,我们可能已经发现某些短词(如「ok」「继续」)既可能是高效推进也可能是疲态;某些词(如「排版」「收官」)可能是收口信号。请基于这些线索深挖。
- 各 session 大小可能差异极大。请确保结论不被某一个 session 主导,要拉所有 session 一起看。

请扫描每个 session 的"过渡点"——从"讨论"切换到"实施"的瞬间。回答:

1. 我用什么显式信号表示"讨论结束、开始实施"?
   请按出现频率排序,列出所有具体句式 + 原文 + session id。
   重点关注:
   - "开干 / 开始实现 / 开工 / 立即开工"等启动词
   - "commit / 合并 / push" 等 git 动作
   - "排版 / 收官 / 这一稿"等"整理结论"信号
   - "继续"在不同语境下的真实语义

2. 讨论结束时有没有"产出物确认"?
   - 我会主动要求 AI 输出 PRD / 设计文档 / 任务清单 / 实施方案吗?
   - 还是我假设 AI 自己会整理?
   - 所有 session 里"明确产出物文档"的比例?

3. 找出"漂亮收尾"vs"草率收尾"的 session,对比。
   - 哪些 session 的收尾是讨论完成 → 自然过渡到实施 → 实施完成?
   - 哪些 session 是讨论一半就跳过去开始编码?
   - 哪些 session 是被中断 / 转向 / 放弃?
   - 中断/转向的触发点是什么?

4. 讨论长度统计:
   - 每个 session 大致跑了多少轮"用户-AI"对话?
   - 这些"讨论"占整个 session 的多大比例?(剩下是实施/工具调用)
   - 我倾向于讨论多长就开始实施?

5. 收敛模式总结:
   - 我是"边讨论边实施"型,还是"完全讨论完才实施"型?证据?
   - 我有没有"讨论到一半引出新问题、回到讨论"的反复模式?
   - 我对"先做最小可用版本 vs 做完整版本"的倾向?

6. subagent 应对规则:
   - 哪些信号应该让 subagent 立刻停止讨论、进入实施?
   - 哪些信号应该让 subagent 继续追问(用户不是在收口而是在跳过)?
   - subagent 应不应该主动建议"我们差不多可以收口了"?如果应该,什么时候?

要求:
- 每个论断必须有 session id + 原文摘录
- 不要让某个高密度 session 单独主导结论,所有 session 都要采样
- 注意第 1/2 轮已发现的信号词语义,本轮要深化不要重复
```

## 第 4 轮 · 补丁式指令(最高优先级轮)

```
继续分析同一批 session。本轮是六轮里最重要的一轮,请投入最高精力。

本轮任务:找出"我反复手动补充的指令"——这些是 subagent 应该自动做、
但目前每次都要手动说的话。每一条都是 subagent 系统提示的金矿。

请扫描所有 session 的全部 user 消息,找出**跨多个 session 重复出现**的指令性
语句。重点不是"出现一次"的指令,是"反复出现"的——重复才是方法论,单次是偶然。

请按以下 5 个类别归纳,每个类别都要列出:

────────────────────────────────────────────────
**类别 1:流程性指令(讨论/确认流程相关)**

例如:"先让我确认再实施"、"分析后让我确认"、"如有疑问请向我确认"、"与我讨论"、
"输出方案让我确认"、"先 review 再写入"、"先不要开始实现"...

请列出:
- 每条指令的具体原话(用引号)
- 出现在哪些 session 哪些位置
- 出现频次
- 这条指令的意图(为什么我反复说?)
- 如果写进 subagent 系统提示,应该用什么措辞?

────────────────────────────────────────────────
**类别 2:范围性指令(限定操作范围)**

例如:"不要改 X"、"只看 X"、"先忽略 X 模型"、"禁止修改任何代码"、"先看现有代码"、
"基于现有 X 模块"、"不要新增 SQL"、"在原来基础上直接修改"...

请列出同样的 5 项信息。

────────────────────────────────────────────────
**类别 3:输出性指令(产出物/输出形式)**

例如:"输出到 X 目录"、"简洁明了"、"禁止大篇幅介绍"、"非功能需求无需过多介绍"、
"符合敏捷开发"、"输出对核心逻辑有用的逻辑"、"无意义描述禁止"...

请列出同样的 5 项信息。特别关注**反复出现的输出风格约束**。

────────────────────────────────────────────────
**类别 4:风格性指令(技术品味/方案偏好)**

例如:"不要过度设计"、"设计太麻烦了"、"配置太多了"、"完全写死"、"先把流程调通"、
"先做最小可用"、"不要新增表"、"复用现有 X"...

请列出同样的 5 项信息。

────────────────────────────────────────────────
**类别 5:环境性指令(技术栈/平台约束)**

例如:操作系统/Shell 约束(Windows/PowerShell vs Linux/bash、命令链式语法差异)、
单体 vs 微服务、特定框架/版本约束、数据库选型(关系型 vs 文档型 vs 字段类型)、
迁移工具、git 工作流约束等。

请列出同样的 5 项信息。特别关注**纠错型环境指令**——
即"AI 出错后,用户告诉它'我是 X 环境,以后请怎么避免'"的瞬间。

────────────────────────────────────────────────
**额外分析(必答):**

A. **互相矛盾的指令**:有没有"A session 说要 X,B session 说要反 X"的?
   如果有,列出来 + 分析在什么场景下用哪种。

B. **节省的输入量估算**:如果这些指令都进了系统提示 / subagent,
   按出现频次估算我下次开新 session 时能节省多少手动输入?
   (粗略字数 + 心智负担评价)

C. **TOP 5 必入 subagent 的指令**:从所有类别里挑 5 条最高优先级,
   按"出现频次 × 不入 subagent 的后果严重程度"排序。
   每条给出 subagent 应该如何措辞规避。

要求:
- 必须列出原文证据(指令的原话 + session id + 大致位置)
- 不要在"出现一次"的指令上浪费篇幅
- 重复 ≥3 次才算"反复出现",才进入分析
- 如果某个类别没有反复指令,直接说"该类别无反复指令"
- 跨 session 验证,不要让单个高密度 session 主导
```

## 第 5 轮 · 走偏与翻车

```
继续分析同一批 session。本轮和第 4 轮同等重要,因为它产出 subagent 的"必须避免"清单。

本轮任务:找出"讨论跑偏的瞬间"——AI 走错了方向、或者我和 AI 没对上频道的时刻。

请扫描所有 session,找出"翻车点"。重点不是抱怨,是**精准定位翻车机制**——
是 AI 的什么决策导致了翻车?用户用什么策略拉回来?

按以下 5 类归纳:

────────────────────────────────────────────────
**翻车类型 1:AI 误解了我的需求**

例如:用户说 A,AI 听成 B,跑了半天才发现方向错。

请举出 2-3 个最典型的例子,每个例子完整还原:
1. 用户的原话(可能含歧义/省略/隐含假设)
2. AI 误解后做了什么(实际方向)
3. 用户的纠正方式(直接说"你误解了"?重述?给反例?)
4. 纠正用了几个回合
5. 这种误解能否被 subagent 系统提示前置规避?(如果能,具体怎么写)

────────────────────────────────────────────────
**翻车类型 2:AI 过度设计**

例如:用户要求 A,AI 给了一个"完整、健壮、灵活"但用户嫌麻烦的方案。

请举出 2-3 个例子,每个完整还原:
1. 用户的原话(可能比较短,容易被 AI 自动"补全"成复杂版)
2. AI 给的过度方案具体包含什么(具体的过度之处:配置项太多?抽象层太多?为未来留太多扩展点?)
3. 用户用什么句式反对的?(直接说"太麻烦"?反问?给极简替代?)
4. 用户的简化方向是什么?
5. AI 在哪一步就应该意识到"这条路是过度的"?signal 是什么?

────────────────────────────────────────────────
**翻车类型 3:AI 偷懒/敷衍**

例如:用户问"分析 X",AI 给了不到位的回答(空泛、缺证据、没扒代码)。

请举出 2-3 个例子,每个完整还原:
1. 用户的提问
2. AI 不到位的回答具体差在哪?(空谈?没引用文件?编造?)
3. 用户怎么施压让 AI 重新做的?
4. 第二次回答和第一次的差距?(说明 AI 第一次确实可以做更好但没做)
5. subagent 怎么在第一次就避免这种"敷衍模式"?

────────────────────────────────────────────────
**翻车类型 4:AI 自作主张做了没要求的事**

特别关注:删除文件、修改无关代码、新增 commit、改配置等。

请举出 2-3 个例子,每个完整还原:
1. 用户的原始要求边界是什么?
2. AI 跨过边界做了什么?
3. 用户的反应(如"为什么要 X?我有说过吗?"类语气)
4. 后果严重程度(数据丢失?浪费时间?需要回滚?)
5. subagent 应该如何识别"我快要跨边界了"?

────────────────────────────────────────────────
**翻车类型 5:我自己表达模糊导致 AI 跑偏**

这一类是用户自己的问题。承认这一类比指责 AI 重要。

请举出 2-3 个例子,每个完整还原:
1. 我说了什么(原文,包括所有模糊点)
2. 模糊在哪?(指代不明?省略关键信息?假设 AI 知道但其实不知道?)
3. AI 怎么处理的?(瞎猜?反问?跑偏?)
4. 如果 AI 主动反问能避免吗?反问什么?
5. subagent 收到这类模糊输入时,应该用什么提问策略?

────────────────────────────────────────────────
**额外分析(必答):**

A. **结构性翻车 vs 偶发翻车**:

哪些翻车是**每次都翻同样的车**?(结构性,必须在 subagent 规避)
哪些是**偶发的**?(可以容忍,不必为偶发个例改 subagent)

判断标准:同一类翻车在 ≥2 个 session 都出现 = 结构性。

B. **拉回成本统计**:

每类翻车,用户用了几个回合才拉回正轨?
哪类翻车的拉回成本最高?这一类应该被 subagent 优先规避。

C. **TOP 5 必须规避的翻车**:

挑出 5 个 subagent 系统提示里必须写"避免 X"条款的翻车。
按"出现频次 × 拉回成本"排序。
每条给出 subagent 应该如何措辞规避。

要求:
- 这一轮专挖问题,不要安慰
- 必须给 session id + 原文摘录
- 不要让某个高翻车密度的 session 单独主导,其他 session 的翻车也要采样
- 关键发现:**结构性翻车** 是这一轮最重要的产出
```

## 第 6 轮 · 隐式标准(最后一轮提炼)

```
继续分析同一批 session。这是六轮的最后一轮,也是最主观的一轮。

本轮任务:提炼"我从未明说但行为反复体现的隐式标准"。

前面五轮我们已经挖了:开场模式、追问触发器、收敛信号、补丁式指令、翻车点。
这些都是**显性方法论**——能从用户的话里直接看出来。

本轮挖**隐性方法论**——从用户的"接受/拒绝/修改/沉默"行为反推他没说出口的标准。

请综合所有 session 的对话数据,回答:

────────────────────────────────────────────────
**维度 1:复杂度偏好**

不只是"喜欢简单"——简单到什么程度?哪里是临界点?

请回答:
```
